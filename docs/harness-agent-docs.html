💡 What Is Codex Harness?

Here's the short version.

🎯 Simple Analogy

Think about what actually happens when you run an AI agent in production. You need the LLM call to work, but you also need it to not cost $500/day, to not loop forever, to remember context from 3 messages ago, and to not crash your app when the API goes down. Codex Harness handles all of that. You write the task. It handles everything else.

🚀 Quick Start

From zero to a running agent in 5 steps.

1

Install dependencies

Requires Python 3.11+ and Docker Desktop.

git clone <repo> && cd HarnessAgent
poetry install
2

Configure API keys

Set at least one provider key. Both are optional if you use a local model.

cp .env.example .env

# Edit .env — set at least one:
ANTHROPIC_API_KEY=sk-ant-...     # Claude
OPENAI_API_KEY=sk-...            # GPT-4o / GPT-5
OPENAI_MODELS=gpt-4o-mini        # which model(s) to use

# Or for fully local (no API key needed):
LLAMACPP_BASE_URL=http://localhost:8080
3

Start infrastructure

Redis, Qdrant, Neo4j, MLflow, Prometheus, Grafana.

docker compose up -d

# Or just the essentials:
docker compose up -d redis qdrant mlflow
4

Start the harness

# Terminal 1 — API server on port 8000
make api

# Terminal 2 — async agent worker
make worker
5

Run your first agent

# SQL agent
curl -X POST http://localhost:8000/runs \
  -H "Content-Type: application/json" \
  -d '{
    "agent_type": "sql",
    "task": "How many users signed up this week?",
    "metadata": {"connection_string": "sqlite:///./data/sample.db"}
  }'

# Code agent
curl -X POST http://localhost:8000/runs \
  -d '{"agent_type":"code","task":"Write a fibonacci function and test it"}'

# Watch live steps (Server-Sent Events stream)
curl http://localhost:8000/runs/{run_id}/steps

The 8 Stages of BaseAgent.run()

1

🔁 Checkpoint Resume

If the run crashed or timed out last time, it picks up from the last saved checkpoint instead of starting over. Saves you from paying for the same LLM calls twice.

2

🧠 Context Preparation

Old messages get trimmed to keep the context window from filling up. For schema lookups, the graph retriever pulls exactly what it needs through a breadth-first search instead of dumping everything in. In practice this means 400 tokens of context instead of 5,000.

3

🔮 LLM Call

The router checks which providers are healthy and picks the best one. If Claude is rate limited, it tries GPT-4o. If that's down too, it tries your local vLLM instance. Every call updates the token budget. If a provider fails 5 times in a row, the circuit breaker kicks in and stops routing to it.

4

🛡️ Safety Check (Output)

Before any tool runs, the response goes through three safety checks. PII gets redacted, toxicity gets flagged, and the content gets checked against the actual retrieved context. If something looks wrong, the run stops here.

5

👤 HITL Gate (if configured)

Some tools need a human in the loop before running. If you've configured a tool to require approval, the agent pauses and waits. You approve or reject through the API. Requests expire after an hour if nobody responds.

6

🔧 Tool Execution

The tool registry checks that the inputs match the schema, runs the safety check, then calls the tool. Code runs inside a Docker container. SQL runs directly against your database (read-only by default). The result goes straight into the conversation history.

7

🧠 Memory Update

The conversation turn gets pushed to Redis. If the agent just ran a SQL query, the tool adds an edge to the knowledge graph: this query used these tables. Next time someone asks something similar, the graph traversal will find that connection.

8

💾 Checkpoint Save (every 10 steps)

Every 10 steps, the current state gets written to a temp file and renamed atomically. If the process dies at step 23, the next run picks up from step 20. Not perfect, but beats starting over.

🧠 Three-Tier Memory System

Three storage tiers, each used for a different reason. Not complicated, just worth understanding.

🔴 HOT

Short-Term Memory (Redis)

Conversation history + scratch pad. Sub-millisecond reads. TTL: 24 hours. Stores last N messages per run_id. Used for: every LLM call's message history.

🔵 WARM

Long-Term Memory (Qdrant / Chroma / Weaviate)

Semantic vector search over long-term facts. Embedded with sentence-transformers. PII-redacted before storage. Used for: "what did this tenant remember last time?"

🟣 STRUCTURED

Knowledge Graph (NetworkX / Neo4j)

Entities and relationships. Table→column edges, query→table edges, error→query edges. Used by Graph RAG for multi-hop reasoning. Gets smarter with every run.

ℹ️

The memory manager scrubs PII before storing anything. Phones, emails, SSNs, card numbers all get masked. This happens automatically before any data hits the vector DB, so you don't have to remember to do it yourself.

💡

You don't need to set up the knowledge graph manually. When the SQL agent starts, it reads the database schema and builds the initial graph. After that, every successful query adds new edges showing which tables it used. The graph just grows on its own as the agent does its work.

🔁 Hermes Self-Improvement Loop

Hermes watches your agent failures, proposes prompt improvements, and optionally applies them. It runs once an hour in the background.

🎯 Plain English

Hermes watches all the agent failures, figures out what went wrong, proposes a prompt fix, then replays the failing tasks to check if the fix actually helps. If the score clears 70%, it either applies the patch automatically or queues it for your review, depending on how you've configured HERMES_AUTO_APPLY.

🔧 Tool System & MCP

The tools your agents can call, how to add new ones, and how MCP servers fit in.

🗃️

execute_sql

Runs SELECT queries. Read-only by default, auto-adds LIMIT if you forgot it, and updates the knowledge graph with the tables it touched.

📋

list_tables

Lists all tables in the database. The SQL agent calls this first, before it writes any queries, to understand what's available.

📐

describe_table

Returns columns, types, constraints, foreign keys, and row count for a table. Used when the agent needs to know the shape of the data before querying it.

🐍

run_python

Runs Python code in a Docker container with 256MB memory and no network access. Stdout and stderr come back in the result.

🔍

lint_code

Runs ruff against Python code and returns issues with line numbers. Useful when the code agent generates something it needs to validate before running.

📁

read_file / write_file

Read and write files inside the agent's workspace. Path traversal is blocked, so the agent can't escape its working directory.

👤 Human-in-the-Loop (HITL)

For tools that are too risky to run automatically. The agent stops, waits for a human decision, then continues or stops based on the response.

⏸️ HITL Pause-and-Resume Flow
sequenceDiagram participant A as 🤖 Agent participant H as 👤 HITLManager participant R as 📬 Redis participant U as 👤 Human participant API as ⚡ API A->>H: request_approval(ctx, "DELETE query", args) H->>R: store ApprovalRequest {status: pending, TTL: 1h} H->>U: notify (webhook / email) Note over A: Agent PAUSES (await_decision) loop Poll every 2s (up to 1h) A->>R: check status alt Pending R-->>A: still pending end end U->>API: POST /hitl/{id}/approve {resolved_by: "alice"} API->>R: update status = approved R-->>A: approved ✅ Note over A: Agent RESUMES, executes the tool alt Rejected U->>API: POST /hitl/{id}/reject API->>R: update status = rejected R-->>A: rejected ❌ Note over A: Raises HITLRejected, run stops end

MLflow Agent Traces

Nested spans: agent_run → llm_call / tool_call / inter_agent. Token counts, latency, cost per span. Compare runs across experiments.

📡

OpenTelemetry

Infrastructure spans: Redis calls, DB queries, HTTP requests. Exports to Jaeger, Tempo, or any OTLP backend.

Prometheus + Grafana

15 pre-defined metrics. Pre-built dashboard: steps/s, token cost, failure heatmap, DLQ depth, circuit breaker states.

📋

Audit Log

Immutable JSONL: who ran what, when, what tools were called, what decisions were made. SHA-256 payload hashing for privacy.

📬

Dead Letter Queue

Runs that exhausted all retries land here. You can inspect them, fix the underlying issue, and replay them. Or leave them for Hermes to learn from.

📡

SSE Step Stream

Watch every agent step in real time: GET /runs/{id}/steps returns a server-sent events stream of StepEvents.

while ctx.is_budget_ok():   # step_count < max_steps, tokens < max_tokens, time < timeout

    history = await self._fit_history(ctx, history)
    # Sliding window: keeps last N messages that fit context window
    # Summarizes older messages if LLM summarizer available

    retrieval = await ctx.memory.smart_retrieve(ctx.task, ctx)
    # Graph RAG: entity extract → BFS → ~400 tokens
    # vs naive vector: ~5,000 tokens — 83% reduction

    # ── Stage 3: LLM call ─────────────────────────────────

        messages, max_tokens=2048, system=self.build_system_prompt(ctx)
    )
    # Router: Claude → GPT-4o → vLLM → llama.cpp (health-aware, circuit-broken)

    ctx.tick(tokens=response.input_tokens + response.output_tokens)
    # Raises BudgetExceeded if over limit

    # ── Stage 4: Safety check ─────────────────────────────
    guard, _ = self._safety_pipeline.check_output({"content": response.content})
    if guard.blocked:
        raise SafetyViolation(guard.reason)

    # ── Done? ─────────────────────────────────────────────
    if not response.tool_calls:
        output = self.extract_final_answer(history)
        break

    # ── Stages 5-7: Tool execution ────────────────────────
    for call in response.tool_calls:
        await self._check_hitl(ctx, call)   # pause if approval needed
        result = await self._tool_registry.execute(ctx, call)
        await ctx.memory.push_message(ctx.run_id, "tool", result.to_text())

    # ── Stage 8: Checkpoint ───────────────────────────────
    if ctx.step_count % 10 == 0:
        await self._checkpoint_manager.save(ctx, history)  # atomic

Step 5: Tool Registry validates and executes

File: src/harness/tools/registry.py

async def execute(self, ctx: AgentContext, call: ToolCall) -> ToolResult:
    tool = self.get(call.name)
    if not tool:
        raise ToolError(f"Unknown tool: {call.name}", FailureClass.TOOL_NOT_FOUND)

    # 1. Validate inputs against JSON Schema
    jsonschema.validate(call.args, tool.input_schema)

    # 2. Safety check via Guardrail intermediate stage
    guard, _ = self._safety.check_step({"tool": call.name, "args": call.args})
    if guard.blocked:
        raise SafetyViolation(guard.reason)

    # 3. Execute with timeout
    async with asyncio.timeout(tool.timeout_seconds):
        result = await tool.execute(ctx, call.args)

    return result
